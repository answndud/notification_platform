## 학습 목표

- ACL 기반 권한 분리를 설계할 수 있다.

## 핵심 원칙

1. 최소 권한
2. 네트워크 격리
3. 암호화 전송
4. 비밀 정보 관리

보안은 "기능 추가"가 아니라 "공격면 축소"입니다.
Redis는 빠르고 단순한 만큼,
기본 보안 통제가 빠지면 침해 영향이 매우 크게 확산될 수 있습니다.

## ACL 개요

ACL은 사용자별 명령/키 접근 범위를 제한합니다.
예: 읽기 전용 사용자, 특정 키 패턴만 허용 사용자.

권장 분리 예시:

- app-readwrite: 애플리케이션 기본 계정
- app-readonly: 조회 전용 배치/리포트 계정
- ops-admin: 운영 점검 계정(강한 통제 하 사용)

핵심은 "역할별 계정 분리"입니다.
하나의 공용 계정은 추적성, 사고 격리, 권한 통제를 모두 악화시킵니다.

## TLS와 네트워크

- 내부망 전용 접근
- 보안 그룹/방화벽으로 접근 출처 제한
- 가능하면 TLS로 평문 전송 방지

추가 권장:

- 인터넷 직접 노출 금지
- 관리 포트는 VPN/점프호스트 경유 제한
- 인증 실패/권한 실패 로그 수집

## 설계 포인트

- 운영자 계정과 애플리케이션 계정 분리
- 민감 명령(`FLUSHALL`, `CONFIG`) 권한 제한
- 비밀번호/인증 정보는 코드 저장소에 커밋 금지

보안 운영 포인트:

- 비밀번호/토큰 주기적 회전
- 계정 미사용 기간 점검 후 비활성화
- 감사 로그를 보관하고 이상 징후 탐지 규칙 운영

## 자주 하는 실수

1. 단일 공용 계정 공유
2. 내부망이라 TLS/ACL을 생략
3. 접근 로그/감사 추적 미구축
4. 운영 편의를 이유로 관리자 권한을 과다 부여

## 요약

- 최소 권한과 비밀정보 관리 원칙이 기본이다.
